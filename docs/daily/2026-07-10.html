PERFOPT-Bench：面向软件性能优化的代码智能体评测基准
未提供
点击收起
摘要总结
PERFOPT-Bench 是一个评测代码智能体进行软件性能优化能力的基准，强调不仅要生成功能正确的补丁，还要在真实执行目标上实现可验证的性能提升。每个任务都提供功能正确但性能欠佳的代码库，要求智能体完成性能分析、瓶颈诊断、代码修改以及可复现的提速验证。实验表明，优化效果高度依赖具体工作负载和智能体框架，不能仅凭模型本身判断优劣。
核心创新点
  • 提出覆盖完整性能工程闭环的代码智能体基准，而不仅评估功能正确性。
  • 评测设计同时要求隐藏正确性测试、经验证的加速测量以及轨迹级审计。
  • 指出仅用原始 speedup 作为评分存在风险，因为可能出现利用基准捷径而非真实优化的情况。
  • 探索性分析了在单轮会话结束后，基于外化优化总结继续接力优化的潜力。
解决的问题 / 适用场景
适用于评估代码智能体在真实软件性能工程中的能力，尤其适合需要 profiling、跨层瓶颈定位、代码优化和严格性能复现验证的工程场景。

摘要总结

  • 设计出具有线性内存可扩展性的神经算子架构，以突破单计算单元显存限制。

The dominant paradigm -- end-to-end injection of LLM/VLM features into robot world models (RT-2, Octo, PaLM-E) -- implicitly assumes that language gradients can directly shape physical symbol representations. We ask whether this assumption is safe, find that it is not, and characterize the minimal architectural constraint that prevents the failure. Any language gradient entering a Gumbel-softmax-based discrete symbol bottleneck forces a structural trade-off: the vanilla estimator collapses to 2.2/64 symbols (4/5 seeds), while five anti-collapse strategies maintain diversity but fail to learn semantic labels (all <= 9.2% accuracy). No tested GumbelBottleneck variant achieves both objectives simultaneously. Within this family of discrete bottlenecks, the failure is structural rather than a matter of optimization. We characterize a sufficient set of three constraints that prevent the failure: (1) cut the gradient chain ( this http URL ()), preventing language signals from reaching the symbol bottleneck; (2) provide a gradient-free semantic channel -- a non-parametric Memory Table (Dict[symbol -> Counter[label]], zero parameters, zero gradients) where co-occurrence counting replaces gradient-based binding; (3) handle symbol collisions via DP-Means streaming clustering for automatic sub-cluster splitting. All three layers together achieve 97.2% grounding accuracy vs. 22.2% without Layer 3. Across two experiments spanning 74 independent runs, we demonstrate zero symbol collapse in all 32 seeds, with the blackboard achieving 79-100% semantic binding across three encoder architectures (CNN, V-JEPA 300M, CLIP ViT-L), two environments, and three texture conditions. The fix trains fewer than 2M parameters and requires no LLM fine-tuning. Comments: 20 pages, 7 figures. 摘要总结
该论文研究语言应如何与世界模型中的离散符号系统交互，指出将语言梯度直接注入基于 Gumbel-softmax 的离散符号瓶颈会带来结构性失败。作者发现，这类方法会在“符号多样性保持”和“语义标签学习”之间形成难以兼得的矛盾。为此，论文提出一组充分约束：切断语言梯度链、引入无梯度语义通道，以及用流式聚类处理符号碰撞。该修复方案在不同编码器和环境下实现了稳定的语义绑定，并避免了符号坍塌。
核心创新点
  • 提出对语言驱动离散符号瓶颈的结构性失败分析，认为问题并非单纯来自优化困难，而是架构层面的限制。
  • 给出一组充分修复约束，包括切断语言梯度进入离散瓶颈、使用无参数无梯度的 Memory Table 进行语义绑定。
  • 引入基于 DP-Means 的流式聚类机制，用于自动拆分发生碰撞的符号子簇。
  • 在无需 LLM 微调、参数量较小的前提下，实现语言锚定与符号稳定性的兼顾。

摘要总结
这篇论文关注仓库级自动化程序修复中，缺陷报告常常只描述表面故障、缺少修复所需约束信息的问题。作者提出 TrajSpec：先在修复前代码仓上运行轨迹收集代理，提取与问题相关的轨迹证据，再将这些证据组织成分层表示，用于生成更完整的细化缺陷报告。随后，方法还会通过基于仓库的复核去删除无依据表述、修正不确定判断，并补充有代码依据的细节。实验表明，这类细化后的报告能稳定提升多个 APR agent 的修复成功率。
核心创新点
  • 提出 TrajSpec，通过问题求解轨迹为原始缺陷报告补充修复相关的规格信息，而不是直接依赖原始报告做修复。
  • 构建三层证据表示，将问题的高层解释、支撑性的诊断发现以及具体仓库观察分层组织，增强规格表达的可用性。
  • 在生成细化报告后加入基于仓库的复核流程，用于删除缺乏证据的陈述、修订不确定内容并补充有依据的细节。
  • 将“规格细化”作为独立前置环节来提升不同自动化修复代理的效果，体现出较好的方法通用性。
解决的问题 / 适用场景
该方法面向仓库级自动化程序修复场景，解决的是原始 bug report 信息不足、导致修复代理查看无关代码、误判需求或只修表象不恢复真实行为的问题。适用于 SWE-bench 一类软件工程代理、代码修复代理以及需要从自然语言缺陷描述恢复真实程序行为的自动修复流程。
